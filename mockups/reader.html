Agents

Last edited Apr 8, 2026 · 12 min read

An agent is a system that uses an LLM to decide its next action in a loop, executing tools and observing results until a task is complete. This page covers the core architecture, evaluation approaches, and practical patterns that have emerged from deploying agents in production.[1]

Architecture

The canonical agent loop is deceptively simple: observe → think → act → repeat. The LLM receives context (previous observations, tool results, user instructions), generates a response that may include tool calls, and the harness executes those tool calls and feeds the results back. The loop terminates when the model produces a final answer or hits a resource limit.

What distinguishes agents from simple chain-of-thought prompting is grounding — the ability to take actions in the real world and observe the consequences. A coding agent that can run tests after writing code is categorically different from one that can only generate code. See Another Page for a deeper treatment of grounding strategies.

“The most important thing about agents is not the LLM — it’s the harness. A great harness with a mediocre model beats a mediocre harness with a great model.”

Tool design

Tools should be idempotent wherever possible. An agent that retries a failed file write shouldn’t create duplicates. Tools should return structured results that the model can parse without ambiguity, and they should fail loudly with actionable error messages rather than returning misleading partial results.

For context on how wikihub implements its own agent-facing tools, see Missing Page — that page is planned but not yet written.

Evaluation

Agent evaluation is hard because success depends on both the final output and the trajectory. A coding agent that produces correct code but takes 47 tool calls to get there is worse than one that does it in 5, even if the end result is identical. The standard approach uses a combination of pass rates and cost-weighted scores:

S = 1 N  Σi=1N ri log2(1 + ci)

where ri is the binary reward for task i and ci is the cost in dollars. This penalizes expensive successes and rewards cheap ones.

Implementation patterns

The following Python snippet shows the minimal agent loop used in our benchmarks. Note how tool results are appended to the message history before each new LLM call: 

pythondef run_agent(task: str, tools: list[Tool], max_steps: int = 50) -> str:
    messages = [{"role": "user", "content": task}]

    for step in range(max_steps):
        response = llm.chat(messages, tools=tools)
        messages.append(response)

        if not response.tool_calls:
            return response.content  # final answer

        for call in response.tool_calls:
            result = execute_tool(call)
            messages.append({
                "role": "tool",
                "tool_call_id": call.id,
                "content": result,
            })

    raise AgentBudgetExceeded(f"hit {max_steps} steps")

Handling context windows

As the message history grows, it eventually hits the model’s context limit. Common strategies include sliding-window truncation (drop the oldest messages), summarization (compress old turns into a summary), and retrieval-augmented replay (index old turns and retrieve relevant ones). Each has trade-offs between coherence, cost, and information loss — see the comparison table below.

Strategy Coherence Cost Info loss
Sliding window Medium Low High
Summarization High Medium Medium
RAG replay High High Low

Practical lessons

After running several hundred agent evaluations across SWE-bench and internal benchmarks, a few patterns emerge reliably:

  • Fewer, better tools beat many narrow tools. Agents get confused by large tool menus.
  • Explicit verify steps after mutations (write file → run tests) dramatically improve pass rates.
  • Cost correlates weakly with quality past a threshold — the cheapest 80% of successful runs are almost as good as the most expensive 20%.
  • System prompts with task decomposition instructions outperform prompts that just describe tools.
